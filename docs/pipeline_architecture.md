# Архитектура Пайплайна Контроля Качества "Kavabanga" (v4 Emo-Cache-Iterator)

## 🚨 КРИТИЧЕСКИЕ ПРАВИЛА (УРОКИ, НАПИСАННЫЕ КРОВЬЮ)
1. **ПРЕФИКСЫ ДАТ СТРОГО ОБЯЗАТЕЛЬНЫ.** При скачивании, парсинге или временном хранении аудиофайлов и транскриптов **КАТЕГОРИЧЕСКИ ЗАПРЕЩАЕТСЯ** сохранять файлы только по времени (например, `14-30-00.ogg`). Файлы ОБЯЗАТЕЛЬНО должны сохраняться с префиксом даты (`2026-04-19_14-30-00.ogg`). Иначе произойдет тихая перезапись файлов, перепутывание дней и смешивание базы данных (что приведет к фатальной порче всей аналитики).
2. **МАТЕМАТИЧЕСКИЙ СДВИГ ВРЕМЕНИ.** При склейке многих оффлайн-файлов в единый дневной лог, время каждого отдельного аудиофайла (секунды от начала дня) ДОЛЖНО аппаратно прибавляться к каждой реплике внутри его транскрипта (`start_sec` + `shift`, `end_sec` + `shift`), чтобы избежать наложения таймкодов файлов разного времени друг на друга.

---

## 1. Обзор Системы
Система "Talk Sensor" (или Kavabanga QA) — это сквозной пайплайн для глубокого семантического и акустического аудита аудиозаписей диалогов между бариста и гостями. 
Архитектура v4 базируется на новейших моделях Сбера **GigaAM v3_e2e_rnnt** (с нативной пунктуацией) и **GigaAM-Emo**, а также итеративном извлечении диалогов через **DeepSeek Context Cache**.

## 2. Структура Данных
Поставщиком "сырых" данных выступает **Яндекс.Диск**. Аудиозаписи (`.ogg`) скачиваются микрофонами из оффлайн-точек каждый час (10:00, 11:00 и т.д.).

---

## 3. Этапы Пайплайна

### Этап 1: Ежечасная Транскрибация (GigaAM v3_e2e_rnnt)
- Как только на Яндекс.Диск поступает новый часовой аудиофайл, пайплайн моментально скачивает его.
- Аудио прогоняется через самую передовую модель **GigaAM v3_e2e_rnnt** (она лучше фильтрует шум, умеет расставлять знаки препинания и выдает точные таймкоды на уровне слов).
- Распознанный текст с таймкодами (со сдвигом времени от начала дня) **добавляется (append)** в единое дневное полотно `daily_canvas_YYYY-MM-DD.txt`.
- Таким образом, в течение дня скрипт постоянно пополняет текстовый лог, не ожидая конца смены.

### Этап 2: Semantic Iterator (DeepSeek Reasoner + Context Cache)
- **Запуск:** Строго в конце дня (после 23:30), когда кофейня закрыта и `daily_canvas` полностью собран.
- Один гигантский файл `daily_canvas` (весь день) разом отправляется в промпт модели **DeepSeek Reasoner**. Использование Prefix/Context Caching делает все обращения к этому огромному тексту крайне дешевыми.
- Запускается автоматический цикл `while True`:
  1. Скрипт просит: _"Найди первый диалог"_.
  2. Модель возвращает текст первого диалога с оригинальными таймкодами.
  3. Скрипт просит: _"Найди следующий чистый диалог строго после [ТАЙМКОД КОНЦА ПЕРВОГО]"_.
  4. Процесс идет до тех пор, пока модель не вернет команду `END`.

### Этап 3: Акустический Эмоциональный Анализ (GigaAM-Emo)
- Для **КАЖДОГО** извлеченного диалога скрипт берет его начальный и конечный таймкоды.
- Скрипт вычисляет исходный файл по сдвигу, вырезает точный кусок аудио и отправляет его в модель **GigaAM-Emo**.
- Модель слушает звук и возвращает вероятности (neutral, positive, angry, sad).
- **ВСЕ** диалоги получают общую акустическую оценку "эмоциональность". 
- Если Emo-модель фиксирует, что вероятность `angry` превышает заданный порог, диалог жестко маркируется тегом **"Конфликт"**. DeepSeek в этом решении не участвует (текст не передает эмоции так, как голос).

### Этап 4: Быстрый Аудит (DeepSeek Chat V3)
- Чистый текст диалога + эмоциональная оценка от GigaAM-Emo отправляются в "дешевую" и быструю модель **DeepSeek Chat V3** (`docs/qa_prompt.md`).
- Модель оценивает бариста ИСКЛЮЧИТЕЛЬНО по рабочим KPI:
  - Приветствие / Прощание
  - Прием заказа / Дублирование
  - Доп. продажа (Upsell) / Лояльность
- Возвращает строго валидированный JSON-отчет.

### Этап 5: Загрузка в БД (Supabase)
- Файлы `FINAL_AUDIT_REPORT_{idx}.json` парсятся, и диалоги загружаются в PostgreSQL базу данных Supabase (`dialogs`). Происходит проверка на дубликаты.
